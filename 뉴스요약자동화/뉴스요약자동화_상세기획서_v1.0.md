## 1. 프로젝트 개요

### 1.1 프로젝트 비전

국내외 금융 뉴스, SNS, 경제지표 발표 등을 실시간으로 수집·분석하여 주식 투자에 영향을 주는 핵심 정보를 자동 요약하고, 사용자 맞춤형 이메일 알람을 제공하는 엔드투엔드(End-to-End) 투자 의사결정 지원 시스템을 구축한다. 기존 v1/v2 파이프라인을 통합하고, harness-100 에이전트 팀 패턴을 적용하여 수집·분석·알림의 전 과정을 자동화한다.

### 1.2 핵심 목표

| # | 목표 | 설명 | 성공 지표 (KPI) |
|---|------|------|-----------------|
| 1 | 실시간 뉴스 수집 | 다채널 (RSS, API, SNS, 공시) 뉴스 수집 자동화 | 수집 소스 15개 이상, 5분 이내 반영률 90% |
| 2 | 지능형 필터링 | 3-Tier 키워드 + 다차원 영향도 스코어링 | 오탐율(False Positive) 15% 이하, 미탐율 5% 이하 |
| 3 | LLM 기반 요약 | 뉴스 1줄/3줄 요약 + BULL/BEAR 방향성 판단 | 방향성 정확도 75% 이상, 요약 품질 ROUGE-L 0.6 이상 |
| 4 | 이메일 알람 자동화 | 긴급/일반/일일 리포트 3단계 이메일 알림 | 알림 지연시간 2분 이내, 이메일 전송 성공률 99% |
| 5 | 종목별 투자 시그널 | 종목/섹터/시장 3-Level 시그널 집계 | 14개 주요 종목 커버리지, 시그널 갱신 주기 30분 |
| 6 | 대시보드 통합 | Streamlit 기반 실시간 모니터링 대시보드 | 페이지 로드 3초 이내, v2 분석 결과 실시간 반영 |
| 7 | 안정적 운영 | 24/7 무중단 파이프라인 운영 | 월간 가용률 99.5%, 장애 자동복구 시간 5분 이내 |

### 1.4 용어 정의

| 용어 | 정의 |
|------|------|
| **NewsItem** | 수집된 뉴스 한 건의 데이터 모델. 제목, 본문, 소스, 분석 결과 등 30+ 필드 포함 |
| **Impact Score** | 뉴스의 시장 영향도를 1~10 스케일로 정량화한 점수 (urgency × scope × certainty × tier) |
| **BULL/BEAR** | 뉴스가 시장에 미치는 방향성. 강세(BULL) 또는 약세(BEAR), 중립 없음 |
| **StockImpact** | 특정 뉴스가 특정 종목에 미치는 영향 (방향 + 강도 + 직접/간접 구분) |
| **MarketVerdict** | 전체 뉴스 기반 시장 종합 판단 (방향 + 확신도 + 분위기) |
| **Alert Rule** | 알림 조건·채널·템플릿·쿨다운을 정의하는 규칙 객체 |
| **TimeSlot** | KST 기반 시간대 구분 — PRE_MARKET / MARKET_HOURS / AFTER_MARKET / GLOBAL |
| **MarketDomain** | 뉴스의 시장 도메인 분류 — EQUITY / BOND / FX / COMMODITY / MACRO / CRYPTO |
| **Tier** | 키워드 중요도 등급 — STRONG(3.0x) / MEDIUM(1.0x) / WEAK(0.7x) |
| **harness-100** | Claude Code용 100개 멀티 에이전트 팀 하네스 프레임워크 |

## 2. 문제 정의 및 연구 동기

### 2.1 현재 금융 뉴스 정보 접근의 한계

**문제 1: 정보 과부하 (Information Overload)**
개인 투자자는 하루 수백~수천 건의 금융 뉴스에 노출되지만, 실제 투자 의사결정에 영향을 주는 뉴스는 전체의 5~10%에 불과하다. 중요 뉴스를 놓치거나 노이즈에 휘둘리는 비효율이 발생한다.

**문제 2: 시간 지연 (Latency)**
RSS 피드 기반 수집은 30분~1시간 주기로 갱신되어, 속보성 뉴스(금리 결정, 실적 발표, 지정학 이벤트)에 대한 대응이 늦다. 장 시작 전 프리마켓 뉴스를 놓치면 시초가 대응이 불가하다.

**문제 3: 분석 편향 (Analysis Bias)**
수동 뉴스 분석은 확증 편향(Confirmation Bias)에 취약하다. 보유 종목에 유리한 뉴스만 선택적으로 인지하고, 불리한 시그널을 무시하는 경향이 있다. 시스템적인 BULL/BEAR 강제 판단이 이를 완화할 수 있다.

**문제 4: 다채널 통합 부재**
뉴스(RSS), 공시(DART), 경제지표(FRED/한은), SNS(X) 등 정보 소스가 분산되어 있어 통합적 투자 판단이 어렵다. 하나의 이벤트가 여러 채널에서 서로 다른 형태로 나타나지만, 이를 연결하는 메커니즘이 없다.

### 2.2 시스템 기반 해결 접근 방식

본 시스템은 다음 접근 방식으로 위 문제를 해결한다:

1. **다채널 실시간 수집** → 15개 이상 소스에서 5분 주기로 수집하여 지연 최소화
2. **3-Tier 필터링 + 다차원 스코어링** → 노이즈 제거 + 영향도 정량화로 정보 과부하 해결
3. **LLM 강제 방향성 판단** → NEUTRAL 옵션 제거로 분석 편향 방지
4. **Alert Rule Engine** → 조건별 자동 알림으로 중요 뉴스 실시간 전달
5. **종목·섹터·시장 3-Level 집계** → 분산 정보를 하나의 투자 시그널로 통합

### 2.3 연구 질문 (Research Questions)

| # | 연구 질문 | 검증 방법 |
|---|----------|----------|
| RQ1 | 다차원 영향도 스코어링(urgency×scope×certainty)은 단순 키워드 카운트 대비 투자 관련 뉴스 식별 정확도를 얼마나 향상시키는가? | A/B 비교 실험 (v1 vs v2 스코어링) |

## 3. 관련 연구 및 기존 시스템 분석

### 3.1 금융 뉴스 감성분석 기술 동향 (2025-2026)

**LLM 기반 접근법의 부상:**
2025-2026년 금융 감성분석 분야는 기존 BERT/FinBERT 기반 판별 모델에서 생성형 LLM(Gemini, GPT-4o, Claude, Llama 3)으로 패러다임이 전환되고 있다. 최신 연구에 따르면, Chain-of-Thought(CoT) 프롬프팅과 Domain Knowledge를 결합한 DK-CoT 전략이 금융 감성분석에서 기존 Fine-tuning 모델을 상회하는 성능을 보인다.

**주요 기법 비교:**

| 기법 | 모델 | 정확도 | 비용 | 특징 |
|------|------|--------|------|------|
| FinBERT | BERT Fine-tuned | 87% | 낮음 | 도메인 특화, 학습 필요 |
| LAPEFT | LoRA + Lexicon | 88.56% | 낮음 | 95% 파라미터 절감 |
| DK-CoT | LLM + CoT | 89-91% | 중간 | Zero-shot, 도메인 지식 주입 |
| SuperICL | LLM + ICL | 90%+ | 높음 | Few-shot, 최고 성능 |
| Gemini 2.5 Flash | Google LLM | 85-88% | 매우 낮음 | 빠른 속도, 저비용 |

**한국어 금융 감성분석:**
한국어 금융 감성사전 KOSELF가 애널리스트 리포트 텍스트에서 긍/부정 빈도를 측정하여 투자 의견을 계량화하는 데 활용되고 있다. 본 시스템은 Gemini 2.5 Flash + 키워드 기반 fallback 이중 전략을 채택한다.

### 3.3 harness-100 에이전트 팀 패턴 분석

본 시스템은 harness-100의 다음 하네스에서 설계 패턴을 차용한다:

| 하네스 | 차용 패턴 | 적용 영역 |
|--------|----------|----------|
| **37-web-scraper** | 5-에이전트 크롤링 팀 (target-analyst → crawler → parser → data-manager → monitor) | 뉴스 수집기 설계 |
| **27-data-pipeline** | ETL 오케스트레이션 + DAG 스케줄링 + 3-Layer 모니터링 | 파이프라인 자동화 |

| **44-market-research** | TAM/SAM/SOM 시장 분석 + 웹 데이터 수집 | 시장 영향도 분석 |
| **82-report-generator** | data-collector → analyst → visualizer → writer → summarizer 리포트 팀 | 일일 리포트 + 이메일 |
| **53-financial-modeler** | 시나리오 분석 + 민감도 분석 | 투자 시그널 집계 |

**공통 설계 원칙 (harness-100에서 추출):**
1. **5-에이전트 팀 구조:** 전문화된 에이전트가 병렬 + 순차 조합으로 협업
2. **File-based 데이터 전달:** `_workspace/` 디렉토리를 통한 구조화된 산출물 관리
3. **최대 2회 수정 라운드:** 검증 에이전트의 크로스 밸리데이션 + 재작업
4. **스케일 모드:** Full Pipeline / Focused / Tool / Review 4가지 실행 모드
5. **멱등성 보장:** 모든 파이프라인 단계가 재실행 가능

### 4.2 아키텍처 설계 원칙

**원칙 1: 모듈 독립성 (Loose Coupling)**
각 레이어는 `NewsItem` 데이터 모델을 통해서만 통신한다. 수집기 추가/제거가 분석 레이어에 영향을 주지 않으며, 알림 채널 추가가 분석 로직에 영향을 주지 않는다.

**원칙 2: 점진적 확장 (Progressive Enhancement)**
기존 v2 파이프라인(main.py)을 핵심으로 유지하면서, 새로운 수집기·알림 채널을 플러그인 방식으로 추가한다. 전면 재작성이 아닌 확장 전략을 취한다.

**원칙 3: 장애 격리 (Fault Isolation)**
단일 수집기 장애가 전체 파이프라인을 중단시키지 않는다. LLM API 장애 시 키워드 기반 fallback으로 자동 전환하고, 이메일 전송 실패 시 재시도 큐에 적재한다.

### 4.3 데이터 흐름 라우팅

| 뉴스 유형 | 수집 경로 | 분석 경로 | 알림 경로 |
|-----------|----------|----------|----------|
| 속보 (금리/전쟁/제재) | RSS + Google News | Full Analysis + LLM | URGENT → 즉시 이메일+텔레그램 |

```python
import feedparser
import asyncio
from typing import List
from models.news_item import NewsItem
from config import RSS_FEEDS_KR, RSS_FEEDS_GLOBAL

async def collect_rss_feeds(sources: str = "all") -> List[NewsItem]:
    """RSS 피드 병렬 수집"""
    feeds = []
    if sources in ("all", "kr"):
        feeds.extend(RSS_FEEDS_KR)
    if sources in ("all", "global"):
        feeds.extend(RSS_FEEDS_GLOBAL)
    
    tasks = [_parse_single_feed(url, name) for name, url in feeds]
    results = await asyncio.gather(*tasks, return_exceptions=True)
    
    items = []
    for result in results:
        if isinstance(result, Exception):
            logger.warning(f"RSS 수집 실패: {result}")
            continue
        items.extend(result)
    
    logger.info(f"RSS 수집 완료: {len(items)}건")
    return items
```

### 5.4 전처리 파이프라인 상세

**Step 1: URL Dedup (URL 중복 제거)**
- MD5 해시 기반 완전 일치 검사
- 동일 기사가 여러 소스에서 수집될 때 첫 수집 건만 유지

**Step 2: Title Dedup (제목 유사도 중복 제거)**
- `difflib.SequenceMatcher` 기반, 임계값 ≥ 0.7
- 같은 이벤트의 재탕/후속 기사 통합

**Step 3: Cache Filter (캐시 필터)**
- JSON 파일 기반 URL 캐시 (`data/cache/url_cache.json`)
- TTL: 24시간 (설정 가능)
- 이전 사이클에서 처리된 뉴스 재처리 방지

**Step 4: Text Clean (텍스트 정제)**
- HTML 태그 제거 (`BeautifulSoup`)
- 유니코드 정규화 (NFKC)
- 연속 공백/개행 정리
- 특수문자 및 광고 텍스트 제거

**Step 5: Body Extract (본문 추출 — 선택적)**
- `trafilatura` 라이브러리로 URL에서 본문 전문 추출
- 타임아웃: 10초/건
- 실패 시 snippet만 사용 (graceful degradation)

### 6.3 키워드 기반 Fallback 분석

LLM API 장애 시 사용하는 규칙 기반 분석:

```python
BULL_KEYWORDS = [
    "호실적", "상승", "급등", "surge", "beat", "upgrade", "bullish",
    "growth", "성장", "흑자", "수혜", "호재", "사상최고", "상향",
    "매수", "outperform", "rally", "breakthrough", "사상최대"
]

BEAR_KEYWORDS = [
    "적자", "하락", "급락", "plunge", "downgrade", "bearish",
    "decline", "loss", "감소", "악재", "리스크", "하향", "위기",
    "매도", "underperform", "crash", "제재", "불확실"
]

def fallback_analyze(text: str) -> dict:
    bull_count = sum(1 for kw in BULL_KEYWORDS if kw in text.lower())
    bear_count = sum(1 for kw in BEAR_KEYWORDS if kw in text.lower())
    
    direction = "BULL" if bull_count >= bear_count else "BEAR"
    total = bull_count + bear_count
    confidence = min(0.95, 0.5 + abs(bull_count - bear_count) / max(total, 1) * 0.4)
    
    return {
        "direction": direction,
        "confidence": confidence,
        "summary_1line": text[:50],
        "investment_signal": f"{'강세' if direction == 'BULL' else '약세'} 키워드 감지",
        "action_suggestion": "관망",
        "risk_factor": "키워드 기반 분석의 한계 — LLM 분석 불가 상태"
    }
```

**Alert Rule 정의:**

| Rule 이름 | 조건 | 채널 | 템플릿 | 쿨다운 | 배치 윈도우 |
|-----------|------|------|--------|--------|-----------|
| 긴급속보 | impact_score ≥ 8.0 | 이메일 + 텔레그램 | `urgent.html` | 30분 | 즉시 (0분) |

```python
# notifiers/alert_engine.py
from dataclasses import dataclass
from typing import Callable, List
from models.news_item import NewsItem

@dataclass
class AlertRule:
    name: str
    condition: Callable[[NewsItem], bool]
    channels: List[str]          # ["email", "telegram"]
    template: str                # 이메일 HTML 템플릿
    cooldown_minutes: int        # 동일 종목 재알림 방지
    batch_window_minutes: int    # 0 = 즉시, N = N분 집계 후 발송

class AlertEngine:
    def __init__(self, rules: List[AlertRule]):
        self.rules = rules
        self.cooldown_tracker = {}   # {stock_name: last_alert_time}
        self.batch_queue = {}        # {rule_name: [NewsItem]}
        self.alert_history = []
    
    def evaluate(self, items: List[NewsItem]):
        """분석 완료된 뉴스 목록에 대해 알림 룰 평가"""
        for item in items:
            for rule in self.rules:
                if rule.condition(item):
                    if self._check_cooldown(rule, item):
                        if rule.batch_window_minutes == 0:
                            self._send_immediate(rule, item)
                        else:
                            self._add_to_batch(rule, item)
    
    def flush_batches(self):
        """배치 윈도우 경과한 큐 발송"""
        now = datetime.now()
        for rule_name, queue_info in self.batch_queue.items():
            if now - queue_info["first_added"] >= timedelta(
                minutes=queue_info["window"]
            ):
                self._send_batch(rule_name, queue_info["items"])
                queue_info["items"] = []
```

```python
import base64
from email.mime.text import MIMEText
from email.mime.multipart import MIMEMultipart
from google.auth.transport.requests import Request
from google.oauth2.credentials import Credentials
from google_auth_oauthlib.flow import InstalledAppFlow
from googleapiclient.discovery import build

SCOPES = ["https://www.googleapis.com/auth/gmail.send"]

class GmailNotifier:
    def __init__(self, credentials_path: str = "credentials.json"):
        self.service = self._authenticate(credentials_path)
    
    def _authenticate(self, credentials_path: str):
        """OAuth2 인증 (최초 1회 브라우저 인증 필요)"""
        creds = None
        token_path = "data/gmail_token.json"
        
        if os.path.exists(token_path):
            creds = Credentials.from_authorized_user_file(token_path, SCOPES)
        
        if not creds or not creds.valid:
            if creds and creds.expired and creds.refresh_token:
                creds.refresh(Request())
            else:
                flow = InstalledAppFlow.from_client_secrets_file(
                    credentials_path, SCOPES
                )
                creds = flow.run_local_server(port=0)
            
            with open(token_path, "w") as f:
                f.write(creds.to_json())
        
        return build("gmail", "v1", credentials=creds)
    
    def send_alert(self, to: str, subject: str, html_body: str):
        """HTML 이메일 전송"""
        message = MIMEMultipart("alternative")
        message["to"] = to
        message["subject"] = subject
        message.attach(MIMEText(html_body, "html"))
        
        raw = base64.urlsafe_b64encode(message.as_bytes()).decode()
        body = {"raw": raw}
        
        sent = self.service.users().messages().send(
            userId="me", body=body
        ).execute()
        
        return sent["id"]
```

| Gradio | O | △ | 기본적 | 쉬움 | 대안 |
| Dash (Plotly) | O | O (callback) | Plotly | 보통 | 고급 대안 |
| FastAPI + React | X (프론트 분리) | O (WebSocket) | 자유 | 복잡 | 규모 확장 시 |

**선정 이유:** 단일 사용자 기준, Python 코드베이스와의 통합, 빠른 프로토타이핑이 우선이므로 Streamlit을 채택한다. 기존 app.py 구조를 확장하여 v2 분석 결과를 반영한다.

## 10. 평가 계획

### 10.1 핵심 성능 메트릭

| 메트릭 | 설명 | 목표값 | 측정 방법 |
|--------|------|--------|----------|
| 방향성 정확도 | BULL/BEAR 판단의 정확도 | ≥ 75% | 수동 라벨링 200건 비교 |
| 영향도 상관관계 | Impact Score vs 실제 주가 변동 | Pearson r ≥ 0.4 | 2주간 뉴스-주가 매핑 |
| 수집 완전성 | 주요 뉴스 누락률 | ≤ 5% | 주요 이벤트 체크리스트 |
| 알림 지연시간 | 뉴스 발생 → 알림 도달 | ≤ 2분 (URGENT) | 타임스탬프 비교 |
| 이메일 전송률 | 알림 이메일 전송 성공 비율 | ≥ 99% | 발송/실패 로그 |
| 오탐율 (False Positive) | 비관련 뉴스를 관련으로 분류 | ≤ 15% | 필터 통과 뉴스 수동 검토 |
| 시스템 가용률 | 파이프라인 정상 가동 비율 | ≥ 99.5% | 헬스체크 로그 |

### 10.2 비교 실험 설계

**실험 1: 스코어링 방식 비교**

| 방식 | 설명 | 비교 대상 |
|------|------|----------|
| v1 (기존) | 단순 키워드 카운트 (1-5) | Baseline |
| v2 (현재) | urgency×scope×certainty×tier → sigmoid 1-10 | 개선안 |

**실험 2: LLM vs 키워드 기반 방향성 판단**

| 방식 | 모델 | 예상 정확도 |
|------|------|-----------|
| 키워드 기반 | BULL/BEAR 키워드 카운트 | 60-65% |
| Gemini 2.5 Flash | Zero-shot 프롬프트 | 80-85% |
| Gemini 2.5 Flash + CoT | Chain-of-Thought | 85-88% |
| FinBERT (참고) | Fine-tuned BERT | 85-87% |

**실험 3: 스케줄링 방식 비교**

| 방식 | 설명 | 중요 뉴스 포착률 |
|------|------|----------------|
| 고정 30분 | 기존 방식 | Baseline |
| 차등 5/15/60분 | 시간대별 차등 | 기대: +20% |

**실험 4: 알림 유효성 평가**

| 메트릭 | 측정 방법 |
|--------|----------|
| 알림 개봉률 | 이메일 open tracking |
| 사용자 반응 시간 | 알림 → 대시보드 접속 시간 |
| 알림 피로도 | 일일 알림 건수 vs 만족도 |

### 10.3 평가 데이터셋 구성

| 유형 | 건수 | 소스 | 용도 |
|------|------|------|------|
| 방향성 판단 검증 | 200건 | 수동 라벨링 (BULL/BEAR) | LLM 정확도 측정 |

## 14. 윤리적 고려사항 및 안전성

### 14.1 핵심 원칙

**원칙 1: 비대체성 (Non-Substitution)**
본 시스템은 투자 의사결정을 **지원**하는 도구이지, 투자 조언 서비스가 아니다. 모든 알림과 리포트에 "자동 생성된 참고 정보이며, 투자 판단의 최종 책임은 사용자에게 있습니다" 면책 조항을 포함한다.

**원칙 2: 근거 기반 (Evidence-Based)**
모든 투자 시그널에는 근거 뉴스 원문 링크를 반드시 포함한다. LLM이 생성한 요약만으로 투자 결정을 유도하지 않으며, 원문 확인을 권장한다.

**원칙 3: 투명성 (Transparency)**
LLM 분석 결과에는 confidence 점수를 명시하여 판단의 확실성 수준을 투명하게 전달한다. Fallback 모드(키워드 기반) 전환 시 사용자에게 명시적으로 고지한다.

**원칙 4: 데이터 최소 수집 (Data Minimization)**
수집하는 뉴스는 공개된 뉴스/공시 정보만 대상으로 하며, 개인정보나 유료 콘텐츠를 크롤링하지 않는다.

```sql
    title TEXT NOT NULL,
    source TEXT NOT NULL,
    source_type TEXT DEFAULT 'RSS',  -- RSS, DART, FRED, BOK, SNS
    url TEXT UNIQUE NOT NULL,
    published_time TEXT,
    collected_time TEXT DEFAULT (datetime('now', 'localtime')),
    region TEXT DEFAULT 'KR',
    language TEXT DEFAULT 'ko',
    full_text TEXT,
    snippet TEXT,
    
    -- 필터링 결과
    keyword_tier TEXT,              -- STRONG, MEDIUM, WEAK
    matched_keywords TEXT,          -- JSON array
    
    -- 스코어링 결과
    impact_score REAL DEFAULT 0,
    urgency REAL DEFAULT 0,
    scope REAL DEFAULT 0,
    certainty REAL DEFAULT 0,
    score_breakdown TEXT,           -- JSON object
    
    -- 분석 결과
    direction TEXT,                 -- BULL, BEAR
    confidence REAL DEFAULT 0.5,
    time_slot TEXT,                 -- PRE_MARKET, MARKET_HOURS, AFTER_MARKET, GLOBAL
    market_domains TEXT,            -- JSON array
    
    -- 종목 영향
    stock_impacts TEXT,             -- JSON array of StockImpact
    tagged_stocks TEXT,             -- JSON array
    
    -- LLM 분석
    summary_1line TEXT,
    summary_3line TEXT,
    investment_signal TEXT,
    action_suggestion TEXT,
    risk_factor TEXT,
    analysis_mode TEXT DEFAULT 'llm',  -- llm, keyword_fallback
    
    -- 인덱스
    created_at TEXT DEFAULT (datetime('now', 'localtime'))
);

CREATE INDEX idx_published_time ON news_items(published_time);
CREATE INDEX idx_impact_score ON news_items(impact_score);
CREATE INDEX idx_direction ON news_items(direction);
CREATE INDEX idx_source_type ON news_items(source_type);

-- 알림 이력
CREATE TABLE IF NOT EXISTS alert_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    rule_name TEXT NOT NULL,
    news_id TEXT REFERENCES news_items(id),
    channel TEXT NOT NULL,           -- email, telegram
    template TEXT,
    status TEXT DEFAULT 'sent',      -- sent, failed, retrying
    sent_at TEXT DEFAULT (datetime('now', 'localtime')),
    error_message TEXT
);

CREATE INDEX idx_alert_sent_at ON alert_history(sent_at);
CREATE INDEX idx_alert_status ON alert_history(status);
```
